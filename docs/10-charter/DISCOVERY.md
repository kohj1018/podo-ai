# Discovery: 채용공고 자동수집 + 신입 개발자 fit/합격가능성 분석 SaaS

## 0. Status
draft

## 1. 문제 한 줄
매일 여러 채널(회사 공식 채용 사이트 등)에 흩어져 올라오는 채용공고·JD를 자동 수집하고, 사용자의 이력서·경력·fit을 분석해 **합격 가능성**과 **fit 적합도**를 점수화·추천하는 SaaS.

## 2. 페르소나

> **현재 근거: 창업자 self-proxy, 외부 인터뷰 전 (가정 — A-6 검증 전).** 아래 역할·맥락·하루 흐름·실패 패턴은 외부 raw 증거(§14)로 검증되지 않은 창업자 직관 기반 가설이다. A-6 인터뷰(외부 신입/졸업예정 개발자 5~8명) 전까지 사실로 취급하지 않는다.

선택된 1순위 페르소나는 "유진"(신입/졸업예정 개발자 구직자, 창업자 self-proxy). 인접 강(强) 페르소나 B(마감 압박형 적극 구직자)·A(재직 중 패시브 탐색자)는 §13 Opportunity Backlog에 보존.

### Persona — "유진" (신입/졸업예정 개발자 구직자)

#### 역할
4학년 2학기 또는 졸업 직후, 첫 정규직을 구하는 신입 개발자. 직무 정체성이 아직 한 줄로 정리 안 됨("개발 쪽은 맞는데 백엔드인지 데이터인지 확신 없음"). 경력 신호 자산이 인턴 1회 + 팀 프로젝트 2~3개 + 자격증/학점 수준으로 한정적.

#### 맥락
수십 개 공고에 동시 지원 중. 채널이 분산돼 있음(잡코리아/사람인/원티드/링크드인/회사 채용페이지/학교 취업포털/오픈채팅). "이 공고에 내 스펙으로 붙을 수 있나?"를 매번 감으로 판단. 신입 공고는 "신입/경력무관" 라벨이 모호하고, 실제로는 우대사항에 경력 요구가 숨어 있어 합격가능성 판단이 가장 어려운 층.

#### 하루 흐름
- (오전) 밤사이 올라온 신규 공고를 채널마다 돌며 수집 → 제목/마감일만 훑고 북마크.
- (오후) 북마크 중 몇 개를 골라 JD 정독, 자소서/이력서 변형.
- (저녁) 지원 완료 기록을 스프레드시트나 메모앱에 수기 관리. 마감 임박 공고를 놓치거나 중복 발견하는 일이 잦음.

#### 현재 해결 방법
채널별 알림 + 북마크 + 직접 만든 지원 트래킹 시트. fit 판단은 "JD 키워드 ↔ 내 이력서 키워드"를 눈으로 대조. 합격가능성은 커뮤니티 후기/취업카페 눈치/주변 합격자 스펙 비교로 추정.

#### 실패 패턴
1. 분산 수집 누락 — 특정 채널만 보다 좋은 공고를 놓침.
2. 과대/과소 지원 — 붙을 데를 겁먹고 거르거나, 안 될 데에 시간 낭비.
3. fit 착시 — JD에 키워드는 겹치는데 실제 요구 수준(연차·스택 깊이)을 못 읽음.
4. 신호 부족으로 점수 신뢰 안 함 — 이력서에 정량 신호가 적어 "분석해줘도 근거가 약하다" 느낌.

#### 측정 가능한 성공 신호
하루 공고 탐색 시간 X분 → Y분 단축; 추천 상위 N개 공고의 실제 지원 전환율(추천→지원); 추천 점수와 실제 서류 통과의 상관(추천 상위군 통과율 > 하위군); 주간 활성 복귀율(매일 오전 알림 클릭률).

## 5. JTBD (Jobs To Be Done)
**주 JTBD (pain #1):** 7개 넘는 채널에 신규 공고가 매일 흩어져 올라올 때, 나는 한 곳에서 빠짐없이 신규·마감 공고를 파악하고 싶다, 그래서 지원 기회를 놓치지 않고 시간을 아끼고 싶다.

**보조 JTBD (pain #2·#3 — fit/합격가능성, F4~F7 도출 근거):** 수집된 공고 중 어디에 지원할지 정해야 할 때, 나는 각 공고가 내 스펙으로 붙을 가능성이 높은지를 근거와 함께 알고 싶다, 그래서 헛지원·과소지원을 줄이고 지원/스킵 의사결정 비용을 낮추고 싶다.

## 6. 시나리오

### Happy path
1. 오전 8시, 전날~당일 새벽 수집된 신규 공고 알림 푸시 수신("신규 12건 / 마감 임박 3건").
2. 앱 진입 → 단일 피드에 수집 채널 공고가 중복 제거되어 통합 노출.
3. 각 공고에 합격가능성(5단계 색깔 밴드)·fit 점수 배지가 함께 표시되어 상위부터 정렬됨.
4. 상위 추천 3~5건을 펼쳐 점수 근거(매칭/gap 요약, JD 원문 인용) 확인.
5. 마감 임박 항목을 별도 섹션에서 확인해 우선순위 조정.
6. "지원하기" 표시로 원본 채널 링크 이동 또는 지원 체크 기록.
7. 처리 완료 공고는 피드에서 정리되어 다음 날 누락 0 상태 유지.

### Alternate path
1. 알림 수신 후 진입했으나 추천 점수가 전반적으로 낮은 날(상위도 보통 밴드).
2. 사용자가 "점수 낮아도 지원하고 싶은" 관심 공고를 수동 즐겨찾기.
3. 시스템은 낮은 점수의 근거(부족 요건)를 명시하되 지원 자체는 막지 않음.
4. 신규 공고가 빈약한 날(2~3건) → "오늘은 적음" 명시 + 최근 7일 미처리 공고 재노출.
5. 직군 미확정 상태 반영: 백엔드/데이터 양쪽 추천을 분리 탭으로 제시.
6. 사용자가 관심 직군 필터를 임시 조정해 피드 재구성.
7. 즐겨찾기·재노출 공고를 합쳐 그날의 지원 후보 리스트를 확정.

### Fail path
<!-- 사용자가 끊을 지점과 수용 가능한 fail을 명시 -->
1. 🔴 **치명** — 오전 알림이 오지 않음(크롤러 지연/실패). *이탈: 앱 존재를 잊음.* 제품 핵심 약속 위반.
2. 🔴 **치명** — 진입했으나 어제와 동일한 공고만 보임(증분 수집 실패). *이탈: "안 도는 앱"으로 신뢰 붕괴.*
3. 🔴 **치명** — 특정 채널 미수집인데 "전부 수집" 인상을 줌(커버리지 미고지). *이탈: 다른 채널에서 놓친 공고 발견 시.* 부분 누락보다 "커버리지 미고지"가 더 치명.
4. 🔴 **치명** — 마감 임박 공고를 놓침(마감일 파싱 오류). *이탈: 실제 기회 손실 1회 발생 즉시.* #1 pain의 본질을 직접 배신.
5. 🔴 **치명(추가, 창업자 명시)** — 점수가 비일관·부정확(틀린 상대 순위·흔들리는 근거). *이탈: 신뢰 즉사.* "근거 없음"보다 "틀린 것"이 훨씬 치명.
6. 🟡 **수용 가능** — 중복 제거 실패로 동일 공고 다중 노출. *이탈: 가독성 저하(기회 손실 X), 반복 시 이탈.*
7. 🟡 **수용 가능** — 점수 근거 없이 숫자/밴드만 표시(단, *틀린* 근거는 아님). *이탈: 추천 무시하고 수동 순회로 회귀(핵심 기능은 생존).*
8. 🟡 **수용 가능(초기)** — 지원 기록 저장 실패 → 다음 날 같은 공고 재노출. *장기 누적 시 치명으로 승격.*

> **수용 가능한 최소 출시선:** 일관성·정확도(GS-1·GS-2)가 프로덕션 수준이면 출시 가능. 중복 노출·근거 빈약·초기 지원기록 누락은 감내. "틀린 점수"는 절대 불가.

## 7. MVP 범위
게이트(일관성·정확도) 최우선 원칙으로, 그 게이트를 실증할 수 있는 최소 묶음만 포함.

**수집 (pain #1)**
- **F1. 회사 공식 채용 사이트 2곳 직접 수집 파이프라인** — 1차 대상 = **토스 · 당근**. 대형 애그리게이터 API가 아닌 회사 공식 채용 페이지 크롤링. (근거: #1 누락 완화 + ToS/안정성 리스크 축소. **A-1 검증됨 — 토스·당근 크롤링 실증 동작 확인, 2026-06-04.**)
- **F2. 커버리지 명시 패널** — "현재 수집: 토스·당근 / 미수집: 나머지"를 UI에 상시 노출. (근거: #1 + 신뢰 게이트 — 거짓 완전성 주장 차단. Fail #3 치명 경로 제거.)
- **F3. 신규/마감임박 변경 감지** — 일자별 diff(신규 공고·D-day). (근거: #1의 "매일×상" 빈도 직격.)

**점수 (pain #2·#3 + 게이트 핵심)**
- **F4. 결정론적·캐시된 스코어링** — 동일 (이력서, JD) 입력 → 동일 점수 보장(LLM 결과 캐시 + temperature=0/고정 seed + 버전 핀). (근거: **일관성 게이트 1순위 장치**, test-retest 변동을 구조적으로 0 수렴.)
- **F5. 상대 랭킹(절대점수 아님 우선)** — 수집된 공고 집합 내 fit 순위화. (근거: #3 + 창업자 명시 "*상대적으로* 어느 게 더 fit한지"가 제품의 핵심. 절대 캘리브레이션보다 순위 정확도가 게이트.)
- **F6. 근거 표시(JD 구절 인용 grounding)** — 각 점수마다 JD 원문 구절을 인용해 "왜 이 점수"를 표시. (근거: #2 숨은 경력요구 가시화 + 정확도 게이트(hallucinated requirement 억제). 근거는 *JD에 실재하는 것만*.)
- **F7. 스코어링 입력 투명성** — 점수 산출에 쓴 입력(이력서 항목 ↔ JD 요구)을 1:1 매핑 노출. **역할: 사용자 노출 기능이 1차**(F6의 "JD 인용 근거"와 구분 — F6은 *JD 측* 요구 출처, F7은 *이력서↔JD 매칭*을 보여 #2·#3의 gap을 사용자가 직접 인지). 동일 매핑이 GS-2/GS-3 측정의 데이터 소스로도 *겸용*되나, MVP 포함 사유는 사용자 노출 가치이며 측정 겸용은 부수 효과다.

**합격가능성 표현 방식**: 정확한 % 숫자가 아닌 **5단계 색깔 밴드**로 표현. (초반 정확도 한계 보완 + 과약속 방지. 밴드 cut-off·폭은 §11 열린 질문 참조.)

## 8. MVP 비범위
- **비개발 직군 / 비개발 JD** — 페르소나가 개발자 단일. 직무 미확정도 개발 내부(백엔드/데이터)로 한정.
- **자소서·이력서 자동작성/첨삭** — 게이트와 직교한 생성 품질 문제. 신뢰 게이트 통과 후 별도 검증.
- **다채널 풀커버리지(7개+ 전부 / 애그리게이터)** — 비목표 유지(무리한 스크래핑 = Fail #3 재발·법적 리스크). ⚠️ **단, M5에서 *공식 채용 페이지* 커버리지를 확대**(토스·당근 → 네카라쿠배·외국계·스타트업 공식 페이지 + 표준 ATS, 티어드, 소스별 A-1형 게이트). **애그리게이터(잡코리아·직행·원티드 등)는 영구 비범위.** F2 커버리지 명시 패널 유지(부분 커버리지 정직 고지). (M5 scope-note, 2026-06-07 — 별도 ADR 불필요: 검증된 A-1 재사용·가역적·"공식만"은 기존 ToS 입장의 연장.)
- **무응답/서류결과 피드백 루프 자동화** — (지원 후 합/불·무응답을 자동 추적해 약한 지원 패턴을 학습시키는 루프, pain #8.) MVP는 수동/소규모 정답셋으로만 측정.
- **절대 합격확률 캘리브레이션(% 보장)** — F5 상대 랭킹 + 5단계 밴드로 대체. 데이터 쌓이기 전 과약속 → 신뢰 즉사 위험.
- **일정/자동지원** — (코테·면접·마감 자동 캘린더 + 원클릭 자동 지원.) 핵심 pain 아님.
- **협업 / 공유** — (취업 스터디·커리어 코치와 *공유*하는 기능.) **비범위 유지.** ⚠️ 단, **멀티유저(각자 OAuth 계정 + 본인 데이터 격리)는 M4 범위로 채택**(ADR-107, 2026-06-07) — "협업/공유"와 구분한다. self-proxy는 *검증 주체가 1인*이라는 뜻이지 제품이 단일계정이라는 제약이 아니다.

## 9. 성공 기준
**GS-1·GS-2 = 출시 차단 게이트(둘 중 하나라도 미달 시 미출시), GS-3 = 출시 후 측정·개선(창업자 결정: 현재 데이터 확보 불가).**

- **GS-1 일관성 — test-retest 점수 안정성 (🔴 출시 게이트)**
  - 측정: 동일 (이력서, JD) 입력을 N=10회 재채점 → 점수 표준편차 / 0~100 스케일(또는 밴드 일치).
  - 임계값: 캐시 hit 시 변동 = **0**(결정론 보장, F4). 캐시 miss 재계산 경로도 **상위 fit 랭킹 top-k 순서 변동 0회**.
  - 게이트 이유: "흔들리는 점수"는 창업자 명시 *치명 fail*. 재현 불가능하면 정확도 논의가 무의미.

- **GS-2 정확도 — 근거 사실성(hallucinated requirement 비율) (🔴 출시 게이트)**
  - 측정: 표본 공고 M개(≥30)에서 표시된 근거 중 **JD 원문에 실재하지 않는 요구를 지어낸 비율** = 지어낸 근거 수 / 전체 근거 수. 사람 1~2인 라벨링.
  - 임계값: hallucinated requirement **≤ 2%** (목표 0%). JD 인용 grounding(F6) 강제로 달성.
  - 게이트 이유: "틀린 근거"는 *치명 fail*. 없는 요구를 지어내면 #2(합격가능성 가늠)를 정면으로 오도.

- **GS-3 정확도 — 상대 랭킹 예측 타당도 (🟡 출시 후 측정 — 창업자 최중요 기준)**
  - 측정: 실서비스 후 실제 서류통과 결과(≥20쌍)를 누적해 추천 **상위군 서류통과율 > 하위군**(또는 Spearman ρ).
  - 임계값(잠정): ρ ≥ 0.5 또는 상위군 통과율 ≥ 1.5× 하위군.
  - 성격: 창업자가 *최중요*로 지목했으나 출시 전 실제 결과 데이터 확보 불가 → **출시 후 측정해 개선 방향에 활용**. (출시 전 선택적 sanity check는 §12 A-3 / 아래 프록시 참조 — 차단 게이트 아님.)
  - **GS-3 출시 전 선택적 프록시(soft, 비차단 — 단 아래 discovery exit check로 연결):** ① 창업자(+가능 시 현업 1~2인)가 동일 이력서 기준 개발 JD 8~10개를 수기 상대 랭킹 → 모델 랭킹과 Kendall τ/Spearman ρ. ② 자명한 페어(명백히 높음/낮음) 10~15쌍의 위반율 ≤5%. 평가자 표본이 작아 통계적 일반화 제한(A-6 연동).
    - **τ 구간별 action (pass/fail 분기 명시):** `τ < 0.6` → F5 상대 랭킹 MVP 포함 여부 + §7 MVP 범위 재검토(A-3 No-go 신호). `0.6 ≤ τ < 0.7` → 조건부 진행 + A-3 재실험(평가자/표본 보강 후 재측정). `τ ≥ 0.7` → 진행.
  - **Discovery exit check (구현 착수 전 게이트 — GS-3을 출시 차단으로 승격하진 않되, MVP 구현 진입 전 1회 통과 요구):** 핵심 thesis(A-3/F5)가 전혀 검증 안 된 채 구현으로 넘어가는 것을 막기 위해, 위 프록시 ①을 *최소 1회* 실행해 `τ ≥ 0.6` 또는 자명 페어 위반율 ≤5% 중 하나라도 충족하지 못하면 구현 착수 전 §7 scope를 재검토한다. (A-3 연동 — 본 게이트가 A-3의 No-go 경로다.)

## 10. 핵심 가정
R0~R2에서 추측·미검증으로 전제된 항목을 §12 Assumption Tracker에 A-1~A-12로 적재. 가장 위험한 Top 3:

1. **A-3 / GS-3 (최고 위험)** — 개발 JD 간 상대 fit 랭킹을 LLM이 일관·정확히 매기는가. 창업자 명시 최중요 기준이나 실제 결과 데이터가 없어 출시 전 정확성 측정 경로 자체가 불확실. 여기 깨지면 F5(상대 랭킹)와 제품 차별화가 붕괴. *검증: 공고쌍 20~30 창업자 수기 랭킹 vs 모델 Kendall τ (수 시간, 코드 최소).*
2. **A-1 (검증됨 2026-06-04 — 리스크 해소)** — 토스·당근 채용 사이트 주기 크롤링이 anti-bot/동적렌더링 위에서 실제로 동작함을 외부에서 직접 확인(차단·캡차 미관측). F1 입력 확보. ToS 준수는 운영 상시 원칙으로 유지(리스크 아님). *검증 완료 → §14 E-1.*
3. **A-6** — self-proxy 1인 검증이 더 넓은 신입 개발자 시장에 일반화되는가. pain·"fit 우선" 전제가 1인 직관뿐이면 정확도 완벽해도 시장이 안 원하는 제품. *검증: 외부 신입/졸업예정 개발자 5~8명 문제 인터뷰.*

## 11. 열린 질문
- GS-3 출시 전 프록시의 전문가 평가자를 창업자 1인 외에 몇 명, 어떻게 확보하는가? 확보 불가 시 soft 신뢰도를 어디까지 낮춰 수용?
- 5단계 색깔 밴드의 경계 cut-off는 무엇이며, 정확도 낮은 초기엔 밴드 폭을 의도적으로 넓게(보수적으로) 잡을 것인가?
- 직군 미확정(백엔드/데이터) 상태에서 단일 스코어링 모델 vs 직군 분기 모델 중 무엇으로 시작? (A-7 결과 의존)
- "숨은 경력요구 추론"의 허용 경계(합리적 inference vs fabrication)를 출시 전 규칙/프롬프트 가드레일로 어떻게 명문화? (R-b 직결)
- 신규/마감 diff(F3)와 fit 점수(F5) 중 사용자 1순위 가치가 인터뷰에서 뒤집힐 경우(A-5/A-8) MVP 기능 우선순위를 재배치할 의향?
- **(P2 반영) A-8 인터뷰에서 "수집 자동화" 가치 자체가 기대 이하(수집이 핵심 pain이 아님)로 판명될 경우, F1~F3의 MVP 포함 여부를 재검토할 준비가 됐는가?** — 현행 §11은 기능 *우선순위 뒤집힘*만 다루나, 수집 가치 자체 반증은 F1~F3 전체 가치 소실로 직결.
- **(P2 반영) 제품이 정상 동작해도 사용자가 이탈하는 시나리오**(가치 자체 불필요 인식, "사람인 앱 알림으로 충분"류 대안 충분)를 A-6 인터뷰에서 별도로 물어 확인할 것인가? — §6 Fail path가 전부 "제품 결함" 방향이라 confirmation bias 보정 필요.

**(P0 반영 — A-3/A-6 pivot 조건을 열린 질문에서 *답변*으로 전환):**
- **A-3 실패 시 pivot 조건:** GS-3 프록시 `τ < 0.6` 또는 자명 페어 위반율 >5% → F5 상대 랭킹의 MVP 포함을 재검토하고 §7 MVP 범위를 재정의한다(상세는 §9 discovery exit check + §12 A-3 다음 행동).
- **A-6 실패 시 pivot 조건:** 외부 5~8인 인터뷰에서 pain #1~#3 공감이 과반 미만으로 재현 실패 시 → 페르소나·핵심 pain을 재정의하고 §7 MVP 범위를 재검토(필요 시 인접 페르소나 B/A로 pivot 검토, §13 연동). 상세는 §12 A-6 다음 행동.
