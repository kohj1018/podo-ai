## 1. 프로젝트 요약
매일 여러 채널(회사 공식 채용 사이트 등)에 흩어져 올라오는 채용공고·JD를 자동 수집하고, 사용자의 이력서·경력·fit을 분석해 **합격 가능성**과 **fit 적합도**를 점수화·추천하는 SaaS. 1순위 사용자는 신입/졸업예정 개발자 구직자다.

제품의 단일 thesis: **"틀린 점수"가 "근거 없는 점수"보다 치명적이다.** 따라서 점수의 *일관성*과 *근거 사실성*이 모든 기능에 우선하는 신뢰 게이트다 (DISCOVERY I-1).

## 2. 대상 사용자
신입/졸업예정 개발자 구직자. 직무 정체성이 아직 한 줄로 정리되지 않았고(백엔드/데이터 사이 미확정), 경력 신호 자산이 인턴 1회 + 팀 프로젝트 2~3개 수준으로 한정적이며, 7개 넘는 채널에 분산 지원 중인 층.

> **신뢰도 표기:** 현재 페르소나는 창업자 self-proxy 기반 가설이며 외부 인터뷰(A-6) 전이다. 사실로 확정된 것이 아니다 (DISCOVERY §2 면책).

## 2.1 페르소나

> DISCOVERY §2에서 박은 1순위 페르소나. 인접 페르소나 B(마감 압박형 적극 구직자)·A(재직 중 패시브 탐색자)는 검증 후 확장 후보로 DISCOVERY §13 Opportunity Backlog에 보존 — 본 MVP 비범위.

### Persona — "유진" (신입/졸업예정 개발자 구직자)

- **역할:** 4학년 2학기 또는 졸업 직후, 첫 정규직을 구하는 신입 개발자. 직무 정체성 미확정("개발 쪽은 맞는데 백엔드인지 데이터인지 확신 없음"). 경력 신호가 인턴 1회 + 팀 프로젝트 2~3개 + 자격증/학점 수준으로 한정적.
- **맥락:** 수십 개 공고에 동시 지원 중. 채널 분산(잡코리아/사람인/원티드/링크드인/회사 채용페이지/학교 취업포털/오픈채팅). "이 공고에 내 스펙으로 붙을 수 있나?"를 매번 감으로 판단. 신입 공고는 "신입/경력무관" 라벨이 모호하고 우대사항에 경력 요구가 숨어 있어 합격가능성 판단이 가장 어려운 층.
- **하루 흐름:** (오전) 밤사이 신규 공고를 채널마다 순회 → 제목/마감일만 훑고 북마크. (오후) 북마크 중 일부 JD 정독, 이력서 변형. (저녁) 지원 기록을 스프레드시트로 수기 관리, 마감 임박을 놓치거나 중복 발견.
- **현재 해결 방법:** 채널별 알림 + 북마크 + 수동 지원 트래킹 시트. fit은 "JD 키워드 ↔ 이력서 키워드" 눈대중. 합격가능성은 커뮤니티 후기/주변 합격자 비교로 추정.
- **측정 가능한 성공 신호:** 하루 공고 탐색 시간 단축; 추천 상위 N개의 지원 전환율; 추천 점수와 실제 서류 통과의 상관(상위군 통과율 > 하위군); 주간 활성 복귀율(오전 알림 클릭률).

## 3. 해결하려는 문제
신입 개발자 구직자는 (1) 7개+ 채널에 매일 흩어진 신규·마감 공고를 직접 순회하다 기회를 놓치고, (2) "신입/경력무관"이라 적혔지만 우대사항에 숨은 경력요구가 있어 합격 가능성을 못 가늠하며, (3) JD 키워드는 겹치는데 실제 요구 수준(연차·스택 깊이)을 못 읽어 fit을 과대평가한다.

세 pain은 "수집 → 합격가능성 → fit"으로 한 줄로 연결되며, 수집(#1)이 나머지의 선행 조건이라 최상위다 (DISCOVERY §4).

> **빈도·고통 면책:** pain의 빈도("매일")·강도("상")는 A-6 인터뷰 전 창업자 추정값이며 외부 사용자 표본에서 관측된 값이 아니다.

## 3.1 핵심 시나리오 (happy / alternate / fail)

### Happy path
1. 오전 8시, 전날~당일 새벽 수집된 신규 공고 알림 푸시 수신("신규 12건 / 마감 임박 3건").
2. 앱 진입 → 단일 피드에 수집 채널 공고가 중복 제거되어 통합 노출.
3. 각 공고에 합격가능성(5단계 색깔 밴드)·fit 점수 배지가 함께 표시되어 상위부터 정렬.
4. 상위 추천 3~5건을 펼쳐 점수 근거(매칭/gap 요약, JD 원문 인용) 확인.
5. 마감 임박 항목을 별도 섹션에서 확인해 우선순위 조정.
6. "지원하기"로 원본 채널 링크 이동 또는 지원 체크 기록.
7. 처리 완료 공고는 피드에서 정리 → 다음 날 누락 0 유지.

### Alternate path
1. 추천 점수가 전반적으로 낮은 날(상위도 보통 밴드).
2. 사용자가 "점수 낮아도 지원하고 싶은" 관심 공고를 수동 즐겨찾기.
3. 시스템은 낮은 점수의 근거(부족 요건)를 명시하되 지원 자체는 막지 않음.
4. 신규 공고가 빈약한 날(2~3건) → "오늘은 적음" 명시 + 최근 7일 미처리 공고 재노출.
5. 직군 미확정 반영: 백엔드/데이터 양쪽 추천을 분리 탭으로 제시.
6. 사용자가 관심 직군 필터를 임시 조정해 피드 재구성.

### Fail path
1. 🔴 **치명** — 오전 알림 미수신(크롤러 지연/실패). 제품 핵심 약속 위반.
2. 🔴 **치명** — 진입했으나 어제와 동일 공고만 보임(증분 수집 실패). "안 도는 앱"으로 신뢰 붕괴.
3. 🔴 **치명** — 특정 채널 미수집인데 "전부 수집" 인상을 줌(커버리지 미고지). 부분 누락보다 "커버리지 미고지"가 더 치명.
4. 🔴 **치명** — 마감 임박 공고를 놓침(마감일 파싱 오류). #1 pain의 본질을 직접 배신.
5. 🔴 **치명** — 점수가 비일관·부정확(틀린 상대 순위·흔들리는 근거). "근거 없음"보다 "틀린 것"이 훨씬 치명.
6. 🟡 **수용 가능** — 중복 제거 실패로 동일 공고 다중 노출(가독성 저하, 기회 손실 X).
7. 🟡 **수용 가능** — 점수 근거 없이 숫자/밴드만 표시(단, *틀린* 근거는 아님).
8. 🟡 **수용 가능(초기)** — 지원 기록 저장 실패 → 다음 날 같은 공고 재노출. 장기 누적 시 치명 승격.

> **수용 가능한 최소 출시선:** 일관성·정확도(GS-1·GS-2)가 프로덕션 수준이면 출시 가능. 중복 노출·근거 빈약·초기 지원기록 누락은 감내. "틀린 점수"는 절대 불가.

## 4. 목표
- **G1.** 분산 채널의 신규·마감 공고를 단일 피드로 누락 없이 제공한다 (pain #1 — 1차 신뢰 계약 "누락 0").
- **G2.** 각 공고에 대해 *재현 가능하고 근거 사실에 grounding된* 합격가능성·fit 신호를 제공한다 (pain #2·#3 — 단일 신뢰 게이트).
- **G3.** 커버리지 범위를 항상 투명하게 고지해 "거짓 완전성" 인상을 차단한다 (Fail #3 제거).
- **G4.** 점수 산출 입력(이력서 항목 ↔ JD 요구)을 사용자에게 1:1로 노출해 gap을 직접 인지하게 한다 (pain #2·#3).

## 5. 비목표
> 범위를 *좁히는* 가장 중요한 절. 다음 plan 라운드의 scope 가드 입력. (DISCOVERY §8)

- **비개발 직군 / 비개발 JD** — 페르소나가 개발자 단일. 직무 미확정도 개발 내부(백엔드/데이터)로 한정.
- **자소서·이력서 자동작성/첨삭** — 신뢰 게이트와 직교한 생성 품질 문제. 게이트 통과 후 별도 검증.
- **다채널 풀커버리지(7개+ 전부)** — F2 커버리지 명시로 대체. 무리한 스크래핑은 Fail #3 재발 + 법적 리스크.
- **무응답/서류결과 피드백 루프 자동화** — MVP는 수동/소규모 정답셋으로만 측정.
- **절대 합격확률 캘리브레이션(% 보장)** — 상대 랭킹 + 5단계 밴드로 대체. 데이터 쌓이기 전 과약속 → 신뢰 즉사.
- **일정/자동지원** — 코테·면접·마감 자동 캘린더 + 원클릭 자동 지원. 핵심 pain 아님.
- **협업 / 멀티유저** — self-proxy 단일 사용자 기준.

## 6. 성공 기준
> **GS-1·GS-2 = 출시 차단 게이트**(둘 중 하나라도 미달 시 미출시). **GS-3 = 출시 후 측정·개선**(현재 데이터 확보 불가). (DISCOVERY §9)

- **GS-1 일관성 — test-retest 점수 안정성 (🔴 출시 게이트)**
  - 측정: 동일 (이력서, JD) 입력을 N=10회 재채점 → 점수 표준편차 / 밴드 일치.
  - 임계값: 캐시 hit 시 변동 = **0**(결정론 보장). 캐시 miss 재계산 경로도 **상위 fit 랭킹 top-k 순서 변동 0회**.
- **GS-2 정확도 — 근거 사실성(hallucinated requirement 비율) (🔴 출시 게이트)**
  - 측정: 표본 공고 M개(≥30)에서 표시된 근거 중 JD 원문에 실재하지 않는 요구를 지어낸 비율. 사람 1~2인 라벨링.
  - 임계값: hallucinated requirement **≤ 2%** (목표 0%). JD 인용 grounding 강제로 달성.
- **GS-3 상대 랭킹 예측 타당도 (🟡 출시 후 측정 — 창업자 최중요 기준)**
  - 측정: 실서비스 후 실제 서류통과 결과(≥20쌍)로 추천 상위군 통과율 > 하위군(또는 Spearman ρ).
  - 임계값(잠정): ρ ≥ 0.5 또는 상위군 통과율 ≥ 1.5× 하위군.
  - **출시 전 선택적 프록시(soft):** 창업자(+가능 시 현업 1~2인)가 개발 JD 8~10개 수기 상대 랭킹 vs 모델 랭킹 Kendall τ. `τ<0.6` → No-go 신호(F5/§범위 재검토), `0.6≤τ<0.7` → 조건부+재실험, `τ≥0.7` → 진행.
  - **Discovery exit check (구현 착수 전 게이트):** 위 프록시를 최소 1회 실행해 `τ≥0.6` 또는 자명 페어 위반율 ≤5% 중 하나라도 충족 못 하면 구현 착수 전 MVP scope 재검토.

## 7. 제약 조건
- **스택 확정 (ADR-101).** TS(web/api)+Python(worker/crawler/eval) 폴리글랏 — Next.js / NestJS+Prisma / Python+OpenAI SDK / Postgres+pgvector. 새 지배 제약은 폴리글랏 schema drift(R6 — ARCH §3-2, schema-contract test로 완화). 인증·푸시 채널은 여전히 미정(단일 사용자 MVP).
- **LLM 비결정성 위의 결정론 요구.** GS-1을 만족하려면 스코어링 출력이 동일 입력에 동일해야 한다 — 캐시 + temperature=0/고정 seed + 모델/프롬프트 버전 핀이 *구조적 전제*(가정 A-12).
- **외부 사이트 의존 (A-1 검증됨 — 리스크 해소).** 토스·당근 공식 채용 페이지 크롤링이 실제 동작함을 외부에서 직접 확인(2026-06-04). F1 입력 확보. ToS 준수는 운영 상시 원칙으로 유지(리스크 아님).
- **단일 개발자 / self-proxy 검증.** 외부 사용자 표본 인터뷰 전 — 전 가정 미검증.
- **출시 전 GS-3 실데이터 확보 불가.** 실제 서류 통과 데이터가 없어 최중요 기준을 출시 전 직접 측정할 수 없다.

## 8. 핵심 사용자 흐름
1. **매일 신규 공고 확인 흐름 (1순위):** 오전 알림 수신 → 단일 피드 진입 → 중복 제거된 신규/마감 공고를 커버리지 패널과 함께 확인.
2. **지원 의사결정 흐름:** 공고별 적합도 5단계·fit 랭킹 확인 → 상위 추천 펼쳐 근거(JD 인용 + 이력서↔JD 매핑) 확인 → 지원/스킵 결정 또는 즐겨찾기.
3. **직군 분기 탐색 흐름:** 직군 미확정 상태에서 백엔드/데이터 추천을 분리 탭으로 비교.

## 9. 핵심 가정
> DISCOVERY §10·§12에서 박은 가장 위험한 Top 3. **A-1 검증됨(2026-06-04)**, 그 외 미검증. (전체 12개는 DISCOVERY §12 Assumption Tracker.)

1. **A-3 / GS-3 (최고 위험)** — 개발 JD 간 상대 fit 랭킹을 LLM이 일관·정확히 매기는가. 창업자 최중요 기준이나 출시 전 결과 데이터 없음. 깨지면 F5(상대 랭킹)와 제품 차별화 붕괴.
   - *검증:* 공고쌍 20~30 창업자 수기 랭킹 vs 모델 Kendall τ (수 시간, 코드 최소).
   - *No-go:* τ<0.6 또는 자명 페어 위반율 >5% → F5 MVP 포함 재검토 + 범위 재정의.
